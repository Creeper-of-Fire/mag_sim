# 001: stdin 泄漏 — 驱动 TUI 的幽灵

**状态**: 已修复
**影响范围**: TUI 界面、模拟进程稳定性
**根本原因**: 子进程启动时 stdin 未重定向，与主进程共享
**修复难度**: 一行代码，排查耗时极长

---

## 现象

TUI 界面会出现以下诡异行为，且完全随机、难以复现：

- 界面窗口状态反复切换（频繁误切换后又切回）
- 输入框中出现大量乱码字符
- 部分按钮、快捷键、鼠标点击、鼠标拖动等操作偶尔被残缺消费，界面看上去卡死（实际是输入事件被截断，未能正确传递）
- 滚动条失去响应
- 模拟程序莫名中断或失败
- 静态测试一切正常，只有模拟跑动时才会触发

这些问题的发生频率和时机完全取决于用户是否在移动鼠标。

## 排查过程

### 阶段一：误判为性能问题

最初怀疑是 TUI 事件循环的性能问题——日志渲染太频繁导致界面卡顿。
尝试了 RichLog 替代方案，发现 RichLog 不支持复制文本，
又通过 issue 找到了 Shift+右键拖动的替代方案，但多栏目处理有问题，最终放弃。

这个阶段一直在日志面板相关的代码上打转，完全没有触及真正的原因。

### 阶段二：错误的解决方向

尝试了两个 Web UI 版本（Vue 和 NiceGUI）来绕开 TUI 问题。
Web 版确实解决了问题（毕竟框架都不一样了），
但引入了大量不必要的复杂度，程序变得臃肿。

### 阶段三：关键线索

一次偶然——当光标聚焦到输入框时移动鼠标，发现输入框里出现了大量乱七八糟的字符。
把这些字符复制给 AI 分析后，才识别出它们是**鼠标操控转义序列**。

这些序列本应被 Textual 框架消费处理，但不知为何泄漏到了输入层。

### 阶段四：真相大白

最终由 GLM-5.1 定位到根因：**启动子进程（模拟进程）时没有重定向 stdin**。

主进程和子进程共享同一个 stdin。当模拟在运行时：
1. 鼠标移动时，终端向 stdin 写入控制序列
2. 由于主进程和子进程共享同一个 stdin，子进程会"截获"部分字节
3. 剩余的残缺序列无法被 Textual 正确解析
4. 残缺的转义序列被当作普通输入，触发各种随机行为

这完美解释了为什么：
- 只有模拟跑动时才出现（子进程此时在运行）
- 症状完全随机（取决于鼠标移动的时机和位置）
- 第一个模拟通常能跑一阵才中断（刚启动时会"愣一下神"）
- 反复拖动滚动条后问题加剧（大量鼠标事件涌入）
- 切换到终端标签页聚焦时也可能触发（具体触发时机随机，取决于鼠标动作）
- 窗口会快速切换又切回（ESC 字符先触发退出又触发返回）

## 根因

子进程共享主进程 stdin，导致鼠标转义序列被部分截获，
产生残缺序列泄漏到 TUI 输入层，引发各种不可预测的行为。

## 修复

重定向子进程的 stdin，一行代码。

## 教训

- 当 bug 与用户物理操作（移动鼠标）耦合时，复现条件极度不稳定，排查难度呈指数增长
- stdin/stdout/stderr 在创建子进程时必须明确处理，不能依赖默认的共享行为
- 有时候关键突破来自于一个完全意外的操作（聚焦到输入框）
- 不要因为一个 bug 就抛弃整个技术栈——先搞清楚真正的原因

## 调试难点

这个 bug 特别难以定位的原因是多方面叠加的：

- **症状与用户行为耦合**：用户移动鼠标的时机和方式完全随机，复现条件不可控
- **发生时机具有欺骗性**：大量错误发生在"切回终端标签页查看"的瞬间，容易误判为后台运行时才出问题，实际上只是那个时候鼠标动了
- **自修复假象**：由于泄漏的字符中含有大量 ESC，窗口状态被误切换后往往会立即切回，看上去像是某种不可预测的错乱
- **静态测试无法发现**：只有模拟在跑动（子进程存活）时才会触发，空转 TUI 完全正常
